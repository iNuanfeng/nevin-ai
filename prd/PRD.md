> 版本：v0.1（MVP）

## 1. 产品定位

一款**个人专属**的 AI 人生管家，以 PWA 形式运行在手机上，部署在自有服务器上。
它知道你的一切——你的背景、价值观、正在经历的事、身边的人——像你的第二大脑，帮你：

- 分析处理复杂人际关系（职场、情场、家庭）
- 规划工作与生活
- 学习成长（摄影等技能方向）

**核心差异**：不是通用 AI 助手，而是**真正记住你和你的社交圈的 AI**。每次对话后自动提炼联系人档案和关键记忆，越用越懂你。

## 2. 核心概念模型

### 2.1 导师（Mentor）—— 创建对话时选择的角色

导师是一个**AI 角色模板**，新建对话时选择。每个导师有独立的人设和系统提示词。

> 导师 = 扮演角色 + 限定思考方式 + 人设定制

用户可以根据需要自定义导师的风格。例如：
- 选择「情场顾问」，要求他：**"讲话风趣痞帅，像童锦程的风格，但在关键时候要沉稳给出真诚建议"**
- 选择「职场军师」，要求他：**"话不多，一针见血，有点毒舌"**

导师的人设配置是**永久的**，会影响该导师下所有对话的回复风格。

**总管家是一个特殊的导师** — 它在导师选择器中默认选中，带「默认」标签，不可隐藏/删除。它的 system prompt 最全面，涵盖工作、情感、家庭等所有领域。就像**全科医生 vs 专科医生**——日常问题找总管家，复杂专项问题找对应导师。

### 2.2 对话（Conversation）—— 每次一个具体问题

首页就是对话列表，平铺展示，按最近使用排序。每条对话显示其导师标签和关联的联系人。

模型：
```
首页（对话列表）
 ├── 小红的暧昧期分析  [情场顾问] 👤小红
 ├── 跟张总的1on1策略  [职场军师] 👤张总
 ├── 下半年职业规划    [总管家]
 └── 小红的生日礼物    [情场顾问] 👤小红
```

**交互流程**：
1. 首页看到所有对话（类似 DeepSeek/ChatGPT）
2. 点击右下角 + → 弹出导师选择面板 → 选导师 → 进入聊天
3. 顶部有导师过滤器 [全部] [总管家] [职场军师] ... 点击筛选
4. 对话可以删除。删除对话**不会**影响联系人的档案。

### 2.3 联系人（Person）—— 共享的通讯录

联系人是一个**全局的、跨对话共享**的人际档案。一旦创建，可以在任意对话中加入使用。

通讯录模型：
```
联系人列表（全局共享）
 ├── 小红  → 朋友/恋人
 ├── 张总  → 领导/导师
 ├── 小王  → 同事/竞争者
 └── 妈妈  → 家人
```

**两种联系人添加方式**：
- **方式一（MVP）**：提前在通讯录创建联系人，在对话时手动添加进来。对话过程中 AI 会自动丰富联系人的人设。
- **方式二（阶段二）**：在对话中提到了某人，要求 AI 整理信息并新建联系人。

**关键规则**：
- 在对话中**添加了联系人**后，AI 会在对话过程中自动分析该人物的性格、行为模式、关系动态，**丰富联系人的档案**
- 在对话中**只是随口提到**某人（如"建刚昨天说了..."），但没有作为联系人添加，则视为临时人物，不存入通讯录
- 联系人的信息在所有包含该联系人的对话中共享和累积

### 2.4 临时人物（Transient Person）

对话中随口提到但没有添加为联系人的人物，只存在于当前对话的上下文中，不会被全局记忆。

### 3.3 核心工作流

#### 3.3.1 日常对话流

```
用户打开 App
    → 看到导师列表（总管家 / 职场军师 / 情场顾问 / …）
    → 选择一个导师 → 看到该导师下的对话列表
    → 选择已有对话继续，或新建对话
    → 新建对话时可以选择"添加联系人"（从通讯录选择）
    → 发送消息（文字/图片）
        → 后端检索相关记忆 + 联系人档案 + 用户档案
        → 拼接 system prompt
        → 调用 DeepSeek 流式输出
        → 展示打字机效果
    → 对话结束后（或异步）
        → AI 自动总结关键信息
        → 写入 memories 表
        → 如涉及已添加的联系人，更新该联系人的档案
        → 如涉及用户档案信息，更新 profile
```

#### 3.3.2 人际关系处理流

```
场景：你和小红暧昧中，某天她发了条微信你不确定怎么回

操作：
1. 在通讯录创建联系人「小红」→ 关系：恋人
2. 选择情场顾问 → 新建对话
3. 在对话中「添加联系人：小红」
4. 上传聊天截图并提问："她这句话是什么意思？"

AI 分析：
    - 揣测小红的心理状态
    - 分析话语背后的意图和潜台词
    - 给出回复策略（多方案）
    - 同步更新小红的联系人档案

下次对话：
5. 选择同一对话继续
    - AI 记得上次的分析和建议
    - 结合新情况给出跟进建议

换个场景：
6. 选择职场军师 → 新建对话 → 添加联系人「小红」
    - 职场军师也知道小红的相关信息（跨导师共享联系人档案）
```

#### 3.3.3 记忆检索与注入流

```
核心原则：重要信息不能因为对话长了或者过去久远就被忘记。

用户档案 → 每次对话全量注入（始终记得你是谁）
联系人档案 → 对话中添加了谁就注入谁的档案（始终记得这个人的事）
历史记忆 → FTS5 检索 + 重要度排序 → Top-10 注入（保证远距离重要信息不被丢）
对话上下文 → 当前对话的近 N 条消息
```

## 4. 联系人系统详解

### 4.1 联系人字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 姓名 | 真实姓名或代号 | 小红、张总 |
| 关系分类 | 同事 / 恋人 / 家人 / 朋友 | 恋人 |
| 领域分类 | workplace / romance / family | romance |
| 背景信息 | 对这个人的基本了解 | 24岁，互联网运营，性格外向 |
| 性格判断 | AI 积累的性格分析 | 偏感性，容易焦虑，需要安全感 |
| 关系动态 | 你和 TA 的关系状态描述 | 暧昧期，互相有好感但没挑明 |
| 最近动态 | 最近发生的事 | 上周一起看了电影，牵了手 |
| 策略笔记 | 跟 TA 相处的有效策略/禁忌 | 不要在她累的时候聊严肃话题 |

### 4.2 联系人生命周期

```
创建联系人（手动） → 在对话中添加 → AI 自动分析 → 丰富档案 → 跨对话共享累积
                                                                 ↓
新对话再次添加该联系人 → AI 加载已有档案 + 新对话分析 → 继续丰富
```

### 4.3 规则要点

- 联系人**不会因为对话被删除而受影响**
- 联系人可以实现**跨导师共享**（职场军师知道小红的事，情场顾问也知道）
- 删除联系人 = 彻底忘记这个人（所有对话不再能检索到这个人）

## 6. 数据隐私与安全

- **全部自部署**：数据留在你自己的服务器上
- **仅调 DeepSeek API**：对话内容会经过 DeepSeek 服务器
- **不上传任何数据到第三方**：除 DeepSeek API 外无其他外部调用

> **隐私说明**：联系人和对话内容会发送给 DeepSeek API 处理。如对隐私极度敏感，后续可切换到本地模型。

## 9. 用户故事

### 场景 A：恋爱分析

> 你在和小红暧昧。你创建了情场顾问的对话，添加联系人「小红」。上周你们约会你很紧张，你问了策略。本周你又遇到新情况，继续同一个对话追问。AI 记得上周的分析，结合新情况给你建议。同时小红的档案在不断丰富。

### 场景 B：跨导师工作关系

> 你发现小红居然是你公司的新同事。你在职场军师下新建对话，同样添加了联系人「小红」。职场军师读取到小红的人设信息（小红的性格特点），帮你分析工作中怎么跟她相处。

### 场景 C：领导关系

> 你在职场军师下创建对话，添加联系人「张总」（你的领导）。上传了和张总的聊天截图。AI 分析张总的意思、潜台词、当前他对你的态度。同时持续丰富张总的联系人档案。

### 场景 D：家庭矛盾

> 你在家庭调解师下创建对话，添加联系人「妈妈」。倾诉了最近和妈妈的矛盾。AI 结合妈妈的性格档案给出沟通建议。

### 场景 E：很久之前的记忆

> 50 条对话前你提到过某个重要事件（比如你生日时的感受）。新对话中 AI 应该能通过记忆检索准确召回这条信息，而不是只记得最近几条消息。

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| DeepSeek API 不稳定 | 设计模型抽象层，后续可切换 |
| 记忆检索不准 | FTS5 + 重要度排序，后续加向量检索 |
| 联系人档案膨胀 | 设计自动摘要压缩机制 |
| 跨导师信息共享混乱 | 联系人档案统一管理，导师分别读取 |
| 缺乏持续使用动力 | 建立足够低的启动成本，AI 主动记住你 |

## 11. 数据灾备

### 策略：App 内一键下载

不依赖任何外部服务。服务器在时，用 `/backup` 页面下载完整备份包到电脑。

```
备份包内容：
├── nevin.db              ← 数据库完整快照
├── uploads/              ← 所有上传图片
├── BACKUP_INFO.txt       ← 元信息（对话数、备份时间）
└── restore.sh            ← 恢复指引

大小预估（使用 1 年）：~100-200MB
下载耗时（家用带宽）：10-30秒
```

**推荐频率**：至少每月下载一次存到电脑/硬盘。配合服务器每日本地备份（保留30天），两个机制互不依赖。

> 阶段二可以支持 OSS 自动备份，但 MVP 阶段手动下载完全够用。

## 13. 对话删除机制

### 核心原则

> **删除的是聊天界面，不是 AI 的记忆。**

对话删除后，该对话中提炼出的所有记忆、联系人档案更新、关键洞察全部保留。再次新建对话时，导师通过记忆检索系统依然能衔接上以往的所有认知。

### 13.1 删除交互流程

```
对话列表页
  └── 左滑某条对话
       └── 露出红色「删除」按钮
            └── 点击 → 弹出确认弹窗

确认弹窗内容：
┌──────────────────────────┐
│  删除此对话？              │
│                           │
│  对话记录将被删除。        │
│  ✓ 联系人信息不受影响      │
│  ✓ AI 记忆不受影响         │
│  ✓ 下次聊同一话题会自动衔接 │
│                           │
│     [取消]    [删除]       │
└──────────────────────────┘

确认后：
  └── 对话从列表消失
       └── 底部出现 toast：
           ┌──────────────────────────────────┐
           │  已删除                           │
           │  对话记录被删除，联系人信息和 AI   │
           │  记忆不受影响  [撤销]             │
           └──────────────────────────────────┘
            └── 3 秒内点击「撤销」 → 对话恢复
            └── 3 秒超时 → 对话标记为已删除
```

### 13.2 技术实现

删除不是物理删除，而是软删除：

| 字段 | 说明 |
|------|------|
| `conversations.deleted` | BOOLEAN DEFAULT 0，标记是否删除 |
| `conversations.deleted_at` | DATETIME，删除时间（用于可选自动清理）|

```sql
-- 软删除
UPDATE conversations SET deleted = 1, deleted_at = NOW() WHERE id = ?;

-- 撤销删除
UPDATE conversations SET deleted = 0, deleted_at = NULL WHERE id = ?;

-- 查询对话列表（默认过滤已删除）
SELECT * FROM conversations WHERE deleted = 0 ORDER BY updated_at DESC;

-- 记忆检索不受影响（不依赖 conversation 是否删除）
SELECT * FROM memories WHERE entities LIKE '%person_5%' ORDER BY importance DESC;
```

### 13.3 关键设计规则

| 规则 | 说明 |
|------|------|
| 记忆不随对话删除 | `memories` 表独立存在，不受 `conversations.deleted` 影响 |
| 联系人不受影响 | `persons` 表独立存在，不受任何对话删除影响 |
| 可撤销窗口 | 3 秒内可撤销，超时后 `deleted_at` 锁定 |
| 数据不物理删除 | 保留物理数据，仅标记删除（后续可加自动清理已删除 > 30 天的旧记录）|
| 新建对话自动衔接 | 通过记忆检索系统自动注入历史相关记忆，无需任何手动操作 |


## 14. 联系人归档机制（取代删除）

### 设计变更

> **联系人没有"删除"操作，只有"归档"。**

归档不是忘记一个人，而是将其移入"已归档"状态，通讯录默认不显示，但 AI 对该人物的所有认知完整保留。

### 14.1 数据库变更

```sql
-- persons 表新增字段
ALTER TABLE persons ADD COLUMN archived INTEGER DEFAULT 0;
```

### 14.2 归档 vs 删除的行为对比

| 数据 | 删除（旧方案） | 归档（新方案） |
|------|-------------|-------------|
| 通讯录列表 | 彻底消失 | 默认不显示，可在"已归档"分类查看 |
| 联系人档案 | 全部清除 | 完整保留（性格分析、关系动态、策略笔记）|
| 关联的对话 | 标签移除 | 标签保留，对话记录完整可回溯 |
| 记忆检索 | 悬空引用 | 正常检索 |
| 新建对话时 | 无法再添加 | 可以搜索并添加，AI 仍记得全部 |
| 取消归档 | 不可逆 | 随时可取消归档，恢复至正常状态 |

### 14.3 API 变更

```diff
- DELETE /api/persons/:id    ← 删除联系人
+ PUT /api/persons/:id/archive  ← 归档联系人（不可删除）
+ PUT /api/persons/:id/unarchive ← 取消归档

- GET /api/persons           ← 联系人列表
+ GET /api/persons?archived=0  ← 默认只显示未归档
+ GET /api/persons?archived=1  ← 查看已归档联系人
```

### 14.4 交互设计

```
通讯录页
  ├── 默认 tab：常用联系人（archived=0）
  ├── 已归档 tab：可查看和管理已归档的人
  │
操作一个联系人：
  └── 左滑或长按 → 菜单 → 「归档联系人」
       └── 确认弹窗：
           ┌──────────────────────────┐
           │  归档小红？               │
           │                           │
           │  ✓ AI 记忆不受影响         │
           │  ✓ 历史对话仍可回溯        │
           │  ✓ 任何时候可取消归档      │
           │                           │
           │    [取消]    [归档]        │
           └──────────────────────────┘
            └── 归档成功后 → toast：
                ┌──────────────────────┐
                │ 已归档 · 可在「已归档」│
                │ 中找到  [撤销]        │
                └──────────────────────┘
```

### 14.5 设计原则

- **一个人真正从你的生活中消失，由你决定——但不是通过一个按钮。**
- 归档是管理注意力，不是抹除记忆。
- AI 不会因为你归档了一个人而忘记 TA。
- 已归档的联系人仍可以随时恢复。
